### 3. CLI Commands

All commands use cobra. Keep it clean and simple.

**nocklock version**
```
NockLock v0.1.0 (dev)
```

**nocklock init**
- Creates `.nock/config.toml` in the current directory with sensible defaults
- If config already exists, prints "Config already exists at .nock/config.toml" and exits
- Prints "NockLock initialized. Config at .nock/config.toml"

**nocklock wrap -- <command>**
- For PR #1, this is a PASSTHROUGH — it just runs the wrapped command with no fencing
- Prints "NockLock v0.1.0 — fences not yet active (coming in PR #3-6)"
- Spawns the child process and forwards stdin/stdout/stderr
- Exits with the child's exit code
- This proves the wrap mechanism works before we add fences

**nocklock config**
- Reads and prints the current `.nock/config.toml`
- If no config, prints "No config found. Run `nocklock init` to create one."

**nocklock log**
- Stub: prints "No fence events recorded. Events will appear here once fences are active."

**nocklock status**
- Stub: prints "No active fenced sessions."

### 4. Config Structure

```go
type Config struct {
    Project    ProjectConfig    `toml:"project"`
    Filesystem FilesystemConfig `toml:"filesystem"`
    Network    NetworkConfig    `toml:"network"`
    Secrets    SecretsConfig    `toml:"secrets"`
    Logging    LoggingConfig    `toml:"logging"`
    Cloud      CloudConfig      `toml:"cloud"`
}

type ProjectConfig struct {
    Name string `toml:"name"`
    Root string `toml:"root"`
}

type FilesystemConfig struct {
    Allow []string `toml:"allow"`
    Deny  []string `toml:"deny"`
}

type NetworkConfig struct {
    Allow    []string `toml:"allow"`
    AllowAll bool     `toml:"allow_all"`
}

type SecretsConfig struct {
    Pass  []string `toml:"pass"`
    Block []string `toml:"block"`
}

type LoggingConfig struct {
    DB    string `toml:"db"`
    Level string `toml:"level"`
}

type CloudConfig struct {
    Enabled  bool   `toml:"enabled"`
    APIKey   string `toml:"api_key"`
    Endpoint string `toml:"endpoint"`
}
```

## Plugin & Review Pipeline

This repo has a STRICT review pipeline. NockLock is a security tool — code quality IS the product.

### Plugins to Use DURING Development

These plugins are installed. Use them as you build, not just at the end:

1. **superpowers** — Run FIRST before writing any code. Plan the approach. `/superpowers`
2. **security-guidance** — Run on ALL code that touches subprocess spawning, file paths, env vars, or network. `/security-guidance`
3. **code-simplifier** — Run after completing each major file. Spawns 3 agents to review and simplify. `/code-simplifier`
4. **frontend-design** — Not needed for PR #1 (CLI only, no UI)

### Pre-Push Quality Gates

Before pushing, run ALL of these in order:

```
Step 1: Tests
  go test ./... -v          (all tests pass)
  go vet ./...              (no warnings)
  go fmt ./...              (code formatted)

Step 2: Code Review Plugin
  /code-review              (5 parallel agents review the code)
  Address ALL findings before proceeding.

Step 3: Security Guidance Plugin
  /security-guidance        (security-specific review)
  Address ALL findings before proceeding.

Step 4: Code Simplifier Plugin
  /code-simplifier          (3 agents simplify and clean)
  Accept improvements, reject anything that changes behavior.

Step 5: PR Review Toolkit
  /pr-review-toolkit        (structured PR review checklist)
```
