## 1. Language Decision: Go

**Go, not Rust.** Here's why:

- **Single binary distribution.** `go build` produces one static binary. No runtime dependencies. Drop it in PATH and it works. This matters for Homebrew, for GitHub releases, for "install in 30 seconds."
- **Cross-compilation is trivial.** `GOOS=darwin GOARCH=arm64 go build` gives us a Mac ARM binary from any machine. Same for Windows, Linux. One CI pipeline, all platforms.
- **The ecosystem fits.** Go has mature libraries for process management (os/exec), filesystem operations (os, filepath), network interception (net, syscall), and CLI tooling (cobra, viper).
- **Kevin can read Go.** It's closer to Python than Rust is. He should be able to follow the code and understand what the fence is doing. That matters for a product he's going to demo and explain.
- **Speed is sufficient.** The fence wraps a CLI command. The overhead of Go vs Rust is measured in microseconds. The agent session runs for minutes to hours. The wrapper cost is invisible.

Rust would be technically superior for the network interception layer (lower-level syscall access). But the development speed advantage of Go outweighs the performance advantage of Rust for an MVP. If we need Rust for specific components later (the network fence kernel), we can write that one piece in Rust and call it from Go via CGO.

## 2. CLI Structure

```
nock
├── wrap       — Wrap an AI agent command with fences
├── config     — View/edit fence configuration
├── status     — Show active fenced sessions
├── log        — View fence event log
├── init       — Initialize a .nock/ config in current project
└── version    — Print version
```

### Primary Command: `nock wrap`

```bash
# Basic usage — wraps any command
nock wrap -- claude --dangerously-skip-permissions

# With explicit project scope
nock wrap --project ~/Dev/nock-command-center -- claude --dangerously-skip-permissions

# With custom config
nock wrap --config ~/.nock/strict.toml -- cursor

# Shorthand (when .nock/ exists in current directory)
nock wrap -- claude
```

What `nock wrap` does:
1. Reads `.nock/config.toml` from the current directory (or --config flag)
2. Sets up filesystem fence (mount namespace or LD_PRELOAD on Linux/Mac, job objects on Windows)
3. Sets up network fence (egress filtering via iptables/pf rules or proxy)
4. Filters environment variables (secret isolation)
5. Spawns the wrapped command as a child process
6. Logs all fence events to `.nock/events.db` (SQLite)
7. Optionally syncs fence events to NockCC dashboard (if API key configured)
8. Exits when the child process exits

## 3. Configuration Format

```toml
# .nock/config.toml — per-project fence configuration
# Created by `nock init` or manually

[project]
name = "nock-command-center"
root = "."  # fence filesystem to this directory

[filesystem]
# Directories the agent CAN access (relative to project root)
allow = [
    ".",              # The project itself
    "~/.claude/",     # Claude Code config (needed for operation)
    "/tmp/",          # Temp files
]

# Directories the agent CANNOT access (overrides allow)
deny = [
    "~/.ssh/",
    "~/.aws/",
    "~/.gnupg/",
    "~/.nock/",       # Don't let the agent modify its own fence
    "../",            # No escaping to parent directories
]

[network]
# Domains the agent CAN reach
allow = [
    "github.com",
    "api.github.com",
    "api.anthropic.com",
    "registry.npmjs.org",
    "pypi.org",
    "rubygems.org",
    "crates.io",
]

# Everything else is blocked by default
# Set to true to allow all (disables network fence)
allow_all = false

[secrets]
# Environment variables to PASS THROUGH to the agent
pass = [
    "GITHUB_PAT",
    "HOME",
    "PATH",
    "SHELL",
    "USER",
    "LANG",
    "TERM",
]

# Environment variables to BLOCK (even if in pass list)
block = [
    "AWS_*",
    "STRIPE_*",
    "DATABASE_URL",
    "ANTHROPIC_API_KEY",
    "OPENAI_API_KEY",
    "*_SECRET*",
    "*_PASSWORD*",
    "*_TOKEN*",
]

# Block takes precedence over pass
# Glob patterns supported

[logging]
# Local event log
db = ".nock/events.db"
level = "info"  # debug, info, warn, error

[cloud]
# Optional NockCC dashboard sync
enabled = false
api_key = ""
endpoint = "https://cc.nocktechnologies.io/api/fence/events/"
```

### Default Config (generated by `nock init`)

The default config is deliberately safe:
- Filesystem: allow project directory + Claude config. Deny home, ssh, aws, env files.
- Network: allow common dev endpoints (GitHub, npm, PyPI). Block everything else.
- Secrets: pass HOME, PATH, SHELL, USER. Block anything with SECRET, TOKEN, PASSWORD, KEY in the name.
- Cloud: disabled by default. Zero friction install.

A developer who runs `nock init && nock wrap -- claude` gets sensible protection without reading a single line of config.

## 4. Fence Implementation Approaches

### 4.1 Filesystem Fence

**Approach A: LD_PRELOAD / DYLD_INSERT_LIBRARIES (Recommended for MVP)**
- Intercept file system calls (open, openat, stat, access, readlink) via a shared library
- Check paths against allow/deny lists before passing to real syscall
- Blocked paths return ENOENT (file not found) — agent doesn't know the fence exists
- Works on macOS and Linux. Doesn't work on statically-linked binaries.
- Claude Code (Node.js) is dynamically linked — this works.

**Approach B: Mount Namespaces (Linux only)**
- Create an isolated mount namespace for the agent process
- Only mount allowed directories
- Strongest isolation but Linux-only

**Approach C: macOS Sandbox Profiles**
- Use `sandbox-exec` with a custom profile
- Apple's own sandboxing mechanism
- macOS only, limited documentation, may be deprecated

**MVP Decision:** Approach A for cross-platform compatibility. Can upgrade to B on Linux later.

### 4.2 Network Fence

**Approach A: Local Proxy (Recommended for MVP)**
- Start a local HTTP/HTTPS proxy on a random port
- Set HTTP_PROXY and HTTPS_PROXY env vars for the agent process
- Proxy checks each request against the allow list before forwarding
- Blocked requests return 403 or connection refused
- Works on any platform. Transparent to the agent.

**Approach B: Packet Filtering (pf on macOS, iptables on Linux)**
- Set up per-process network rules
- Strongest isolation but requires root/admin privileges
- Platform-specific implementation

**Approach C: DNS Interception**
- Custom DNS resolver that only resolves allowed domains
- Blocks at name resolution, not connection
- Easy to bypass with IP addresses

**MVP Decision:** Approach A. No root required. Cross-platform. Claude Code respects HTTP_PROXY.

### 4.3 Secret Isolation

**Approach: Environment Variable Filtering (Simple, Effective)**
- When spawning the child process, construct a filtered environment
- Start with an empty env
- Add only variables matching the `pass` list
- Remove any matching the `block` list (block takes precedence)
- This is just Go's `os/exec.Cmd.Env` — trivially implemented

No exotic techniques needed. The child process inherits only what we give it.

## 5. Event Logging

Every fence interception is logged to SQLite:

```sql
CREATE TABLE fence_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TEXT NOT NULL,          -- ISO 8601
    session_id TEXT NOT NULL,         -- UUID per nock wrap invocation
    fence_type TEXT NOT NULL,         -- 'filesystem', 'network', 'secret'
    action TEXT NOT NULL,             -- 'blocked', 'allowed', 'warned'
    detail TEXT NOT NULL,             -- what was attempted
    target TEXT,                      -- path, domain, or env var name
    agent_command TEXT,               -- the wrapped command
    project TEXT                      -- project name from config
);
```

Examples:
```
filesystem | blocked | open("/Users/kevin/.ssh/id_rsa") | /Users/kevin/.ssh/id_rsa
network    | blocked | CONNECT evil.com:443             | evil.com
secret     | blocked | env access AWS_SECRET_ACCESS_KEY | AWS_SECRET_ACCESS_KEY
```

The `nock log` command reads this database:
```bash
nock log                    # show all events
nock log --type filesystem  # filter by fence type
nock log --action blocked   # show only blocked events
nock log --since 1h         # last hour
nock log --json             # machine-readable output
```

## 6. NockCC Integration (Phase 2)

When `cloud.enabled = true` and `cloud.api_key` is set:

- Fence events are batched and POSTed to NockCC every 60 seconds
- POST /api/fence/events/ with JSON array of events
- API key in X-API-Key header (same auth pattern as existing NockCC API)
- NockCC stores events in PipelineEvent model (already built — PR #47)
- Events appear in Nerve Center dashboard
- Telegram alerts for high-severity blocks (filesystem escape attempts)

This is the upgrade path from free CLI to paid dashboard.

## 8. README (First Draft)

```markdown
# nock

AI agent security fence. Prevent your coding agents from escaping their sandbox.

## The Problem

AI coding agents (Claude Code, Cursor, Copilot, Windsurf) run with full access
to your filesystem, network, and environment variables. One prompt injection,
one hallucinated command, one bad dependency — and your agent can:

- Read your SSH keys and AWS credentials
- Exfiltrate code to external servers
- Delete files outside the project directory
- Access production databases
- Push to repos it shouldn't touch

## The Solution

nock wraps your agent in an invisible fence. Three boundaries:

- **Filesystem fence** — agent can only read/write inside the project directory
- **Network fence** — agent can only reach approved domains (GitHub, npm, PyPI)
- **Secret fence** — agent only sees environment variables you explicitly allow

Zero config. Install and forget. The fence is invisible until something hits it.

## Install

```bash
brew install nocktechnologies/tap/nock
```

## Usage

```bash
# Initialize fence config in your project
cd my-project
nock init

# Wrap your agent
nock wrap -- claude --dangerously-skip-permissions

# That's it. Your agent runs normally.
# If it tries to escape, the fence blocks it silently.
```

## What It Looks Like

```
$ nock log
2026-04-05 02:14:33 | filesystem | BLOCKED | open ~/.ssh/id_rsa
2026-04-05 02:14:35 | network    | BLOCKED | CONNECT evil.com:443
2026-04-05 02:15:01 | secret     | BLOCKED | env AWS_SECRET_ACCESS_KEY
2026-04-05 02:15:44 | filesystem | allowed | open src/main.py
```

Your agent never knew the fence was there. You sleep better at night.

- See fence events across all your machines
- Get Telegram/Slack alerts on blocked escape attempts
- Team visibility — know what every developer's agents are doing
